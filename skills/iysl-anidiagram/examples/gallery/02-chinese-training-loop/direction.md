# Direction

## Claim

資訊沒有先整理清楚時，團隊會用更多會議補救，而更多會議讓資訊更散、返工更多——這個循環一圈比一圈嚴重。

## Relation

Loop / cycle。每一步都把下一步推得更強，最後回到新的起點；不是一次性的事件序列。

## Audience

內部訓練情境的團隊成員與主管：需要辨識出「補救式開會」是循環的一環，而不是解方。

## Fact List

| # | 事實 | 標記 |
| --- | --- | --- |
| F1 | 原始問題是需求、風險或決策沒有先被整理成可判斷的內容 | must-keep |
| F2 | 主管因為感到不安，傾向加開更多同步會議確認進度 | must-keep |
| F3 | 更多會議讓資訊散落在口頭對話與零碎訊息中 | must-keep |
| F4 | 執行者要回頭撈決策、補紀錄、重新確認，返工增加 | must-keep |
| F5 | 因為返工變多，下次會更早開會，循環被強化 | must-keep |
| F6 | 真正整理、對齊、定義決策的時間越來越少 | must-keep |
| F7 | 這是會反覆加劇的溝通循環，不是一次性的事件列表 | must-keep |

## Animation Story

一個光點沿封閉的環永不停止地循環，行經每個節點時該節點亮起——動畫本身就是論證：這件事沒有終點，補救動作會回頭加重原始問題。對照 `references/animation-semantics.md` 的 Loop / cycle 條目：連續循環、無可見起訖斷點；任何「跑完就結束」的線性動線都是反模式。
