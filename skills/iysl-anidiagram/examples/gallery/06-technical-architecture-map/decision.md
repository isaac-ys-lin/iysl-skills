# Decision

## Source Type

Technical system brief describing many inputs, a shared processing core, a decision gate, and packaged outputs.

## Primary Claim

The system should be understood as a grouped map of connected parts, not as one narrow procedure.

## Relation

System map with multiple inputs, clustered processing, gating, and output packaging.

## Layout

`architecture`

## Why This Fits

This source needs density and grouping. Readers must see several sources feeding a core area, then a decision gate, then packaged outputs. The architecture layout carries this many-to-core-to-output structure without flattening it into a simplistic step list.

## Rejected Alternatives

`flow` would underrepresent parallel sources and clustered processing. `timeline` would wrongly suggest strict chronology. `composite` could work for several separate claims, but here the real need is one connected system map on a single canvas.

## Style And Animation

Use a technical tone with clear zones and restrained contrast. Animation should guide attention from sources into the core and then through the decision gate, but whitespace and grouping should already make the static composition understandable.

## Validation

Confirm the brief calls for a dense map with grouped regions. The spec may use the existing architecture field family, but it must use neutral, original system labels from this repo case, avoid copied vendor or template wording, avoid reusing the legacy asset's exact labels as content, and pass `--verify --check` while remaining readable as a static PNG.
