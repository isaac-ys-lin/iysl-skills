# Direction

## Claim

Review is not a straight line: one decision point routes every change to pass, revise, or escalate — and all paths rejoin at merge.

## Relation

Branching / decision, with a retry loop-back and a rejoin. The reviewer's judgment determines the path; revision flows backward; escalation detours and returns.

## Audience

Contributors and reviewers learning the team's review process: they need to know the default route first, then where judgment changes the path.

## Fact List

| # | Fact | Tag |
| --- | --- | --- |
| F1 | Draft change is prepared for review | must-keep |
| F2 | Reviewer checks clarity, correctness, and risk | must-keep |
| F3 | Low-risk changes can pass directly | must-keep |
| F4 | Issues send the work back for revision and resubmission | must-keep |
| F5 | High-risk or unclear changes escalate to deeper review before approval | must-keep |
| F6 | Approved work is merged and announced | must-keep |
| F7 | The path can branch, loop, and rejoin — it is not a simple sequence | must-keep |

## Animation Story

The three outcomes light up in turn, one per phase of the loop: first the default pass route straight to merge, then the revise route with a token flowing backward along the retry edge to the draft, then the escalation detour through deeper review and back up into merge. The review diamond pulses each time a token reaches it — the decision point is where every story turns. Per the Branching / decision row of `references/animation-semantics.md`: branch paths light up in turn, the decision node gets the emphasis pulse, and retry edges animate as backflow toward the earlier node. Animating all branches simultaneously would hide the either/or.
